# Аудит прототипа v0.3 (2026-07-13)

Статус на момент аудита: 5/5 pytest проходят, веб-панель запускается (HTTP 200),
`fapi.binance.com` и `demo-fapi.binance.com` доступны. Прототип пригоден как база,
но не готов к Testnet-исполнению без исправлений ниже.

Сверка с официальной документацией Binance (developers.binance.com, июль 2026):

- Testnet/Demo REST база `https://demo-fapi.binance.com` — **корректна** (старый
  `testnet.binancefuture.com` заменён demo-режимом).
- С 2025-12-09 условные ордера (STOP_MARKET / TAKE_PROFIT_MARKET и др.) размещаются
  **только** через `POST /fapi/v1/algoOrder` (`algoType=CONDITIONAL`, `triggerPrice`);
  попытка через `/fapi/v1/order` возвращает `-4120 STOP_ORDER_SWITCH_ALGO`.
  Прототип использует правильный эндпоинт.

## Critical

| # | Где | Проблема | Последствие |
|---|-----|----------|-------------|
| C1 | `binance_client.new_algo_order` | `priceProtect: "TRUE"` — документация задаёт enum `true/false` (нижний регистр); нет `clientAlgoId`, нет обработки/различения кодов ошибок Binance | Риск отклонения защитного SL и незащищённой позиции; невозможно идемпотентно повторить постановку |
| C2 | `binance_client.new_market_order` | Нет `newClientOrderId` | Повторный запуск / retry после таймаута может открыть дублирующую позицию — прямое нарушение критерия MVP |
| C3 | весь проект | Нет engine-слоя: нет reconciliation с биржей, нет atomic entry→SL/TP, нет emergency close, нет daily loss lockout, нет восстановления после рестарта | Testnet-торговля в принципе невозможна безопасно |
| C4 | `binance_client._request` | Нет retry/backoff и обработки 429/418 (ban), нет синхронизации server time (при дрейфе часов все signed-запросы падают с -1021) | Обрыв цикла торговли в самый неподходящий момент |
| C5 | `config.Settings` | Значения полей читаются из env **в момент импорта класса** (default в dataclass), а не при создании экземпляра; `validate()` не вызывается в web-путях до `replace()` | Изменение .env без перезапуска процесса игнорируется; несогласованные параметры проходят молча |

## High

| # | Где | Проблема | Последствие |
|---|-----|----------|-------------|
| H1 | `backtest.py` | Дублирует скоринг из `strategy.py` (copy-paste). Уже расходятся: в backtest нет NaN-guard на `atr` | Backtest тестирует не ту стратегию, которая пойдёт в Testnet |
| H2 | `backtest.py` | Выход по SL/TP всегда по цене стопа, без учёта гэпа: если свеча открылась ниже стопа (LONG), реальный выход — по open, хуже стопа | Систематически заниженные убытки → завышенная доходность |
| H3 | `backtest.py` | Не моделируются funding fees (для intraday-позиций через 8h-метки это реальные деньги) | Завышенная доходность |
| H4 | `backtest.py` | Не соблюдаются согласованные правила риска: max 3 сделки/день, дневной лимит убытка 1.5% | Backtest измеряет другую систему, не ту, что будет торговать |
| H5 | данные | Максимум 1500 свечей одним запросом (~15 дней на 15m); нет пагинации, кэша, выбора диапазона дат | Статистически ничтожная выборка; выводы о стратегии недостоверны |

| H7 | `web_app.py` | `symbol` вставляется в HTML без экранирования (валидация — только `endswith("USDT")`): `<SCRIPT>...USDT` проходит | XSS в панели (локальной, но панель планируется на VPS) |
| H8 | `run_backtest.py` / `web_app.py` | `slippage_rate` применяется дважды к входу: цена входа уже сдвинута на slippage, и затем ещё раз вычитается `entry_slip` от notional | Двойной учёт проскальзывания на входе — искажение результатов (консервативное, но неверное) |

## Что уже сделано корректно

- Консервативное правило «SL раньше TP в одной свече» в backtest.
- Вход на open следующей свечи (нет look-ahead в точке входа).
- `merge_asof(direction="backward")` по `close_time` — 1h-тренд не подглядывает в будущее.
- Изоляция секретов: `.env` в `.gitignore`, секреты не выводятся в UI.
- Реальная торговля заблокирована на уровне `Settings.validate()`.
- `Decimal`-квантование tick/step в `market_rules`.
